# アーキテクチャ決定記録

## ADR-001: フレームワーク選定
- 決定: Hono (バックエンド) + React 19 + Vite (フロントエンド)
- 理由: 単一ページ・認証なし・DBなしのMVPにNuxt 3はオーバースペック（SSR/auto-import/ファイルベースルーティング等が不要）。Honoは軽量・高速でAPIプロキシに最適。@hono/vite-dev-serverでReact SPAとのHMR統合が可能。マルチランタイム対応（Node.js/Bun/Cloudflare Workers）で将来のデプロイ先変更にも強い。
- 比較検討: Elysia（Bun依存がリスク）、Fastify（フルスタック統合が弱い）、Express（設計が古い）
- 日付: 2026-03-24

## ADR-002: 認証方式
- 決定: 認証なし（APIキーは環境変数管理）
- 理由: MVPフェーズではシングルユーザー・社内利用前提で十分。認証はv2以降で検討。
- 日付: 2026-03-24

## ADR-003: APIラッパーのStrategy Pattern採用
- 決定: STT/LLM/TTSそれぞれにStrategy interfaceを定義し、ファクトリで実装を切り替え
- 理由: 各レイヤーのAPI切り替えが最重要要件。interfaceを統一することでテスト容易化・新規API追加時の影響範囲を最小化。
- 日付: 2026-03-24

## ADR-004: 音声転送方式
- 決定: フロント→サーバーはmultipart/form-data、サーバー→フロントはJSON（Base64音声）
- 理由: MVPではシンプルさを優先。WebSocket/ストリーミングはv2で検討。Base64はJSONレスポンスに統合でき、メタ情報（transcript, usage等）と一括返却可能。
- 代替案: WebSocket双方向通信（レイテンシ改善だが実装コスト大）
- 日付: 2026-03-24

## ADR-005: 状態管理
- 決定: React hooksによるローカル状態管理（Redux/Zustand不使用）
- 理由: 単一ページアプリでグローバル状態が少ない。useChat / useApiSelection / useAudioRecorder の3つのカスタムhookで十分管理可能。
- 日付: 2026-03-24
